Documentation

<MANO> - Manuscripts Online is a platform designed for teaching and learning digital manuscript studies. It provides a lightweight environment where users can create TEI records for manuscripts, persons, places, and works, browse them in a shared data collection, and share teaching materials, without requiring installations, accounts, or institutional infrastructure. <MANO> is organized into a set of modular tools that support complementary aspects of manuscript description work. Each module can be used independently or as part of an integrated teaching workflow.

How to cite <MANO>. Cite the platform as: MANO – Manuscripts Online, https://mano-project.github.io.

<MANO> is a platform for working with digital manuscript materials directly in the browser. It brings together tools for creating TEI records, browsing the shared data collection, and accessing teaching resources. Because everything runs client-side, users can work with their own files locally, and nothing is stored on a server unless they choose to publish it in a GitHub repository. The platform is intended for both teaching and introductory research contexts. Students can learn TEI, explore manuscript structures, and create descriptions without needing specialised software, while instructors can assemble and share course materials through a simple static repository.

It was created to address a key challenge in manuscript and TEI pedagogy: many existing tools rely on complex infrastructures, licensed editors, or institutional hosting. These environments are powerful but often inaccessible to students, early-career researchers, or instructors working in low-resource or short-term teaching contexts. <MANO> lowers these barriers by offering a lightweight, sustainable alternative that runs entirely in the browser.

The platform is built around a set of guiding principles that shape both its functionality and its technical design:

  • Browser-based and installation-free. All tools run entirely in the browser so that learners can begin working immediately, without setup tasks or user accounts.
  • Serverless and sustainable. The platform avoids databases, backends, or institutional servers. Static hosting on GitHub ensures long-term accessibility with minimal maintenance.
  • Lightweight and modular. Each component functions independently, allowing instructors to use only what they need.
  • Pedagogically driven. The interface and workflows are designed to support teaching scenarios: introducing TEI, exploring manuscript structure, and guiding students through description tasks.
  • Open and reusable. All materials—code, documentation, and example files—are openly available. Through GitHub, users can fork, adapt, and contribute to the shared collection of TEI records.

<MANO> is organized into a set of independent, browser-based tools that together support common tasks in digital manuscript work:

  • Resources: Displays teaching materials hosted in a GitHub repository. Instructors upload resources; learners download them directly from the interface.
  • Editors — Manuscript, Person, Place, and Work: form-based environments for creating structured TEI records without writing XML. Each editor exports a single TEI-XML file.
  • Data Collection: Browses the project's TEI records — manuscripts, persons, places, and works — rendering each record natively from its TEI file directly in the browser.

Each component works autonomously, but together they form a complete workflow for description, browsing, and sharing.

What it is

The Resources section provides access to a shared repository of teaching materials related to digital manuscript studies. These materials are stored in a public GitHub repository. All learners can freely download them from the <MANO> interface, and instructors can contribute new materials to enrich the shared collection.

The Resources repository is open to instructors who wish to share teaching materials with the <MANO> community. By uploading materials to the GitHub repository, you agree to share them under the Creative Commons Attribution–NonCommercial 4.0 (CC BY-NC 4.0) license. This means:

  • materials may be freely reused for educational and non-commercial purposes
  • the original author must always be acknowledged
  • commercial reuse is not permitted

The interface automatically displays contributor's name, contributor's identifier (if available), and learning materials with direct download.

When to use it

Use the Resources section when:

  • learning TEI and manuscript studies,
  • accessing exercises, guides, or teaching materials contributed by others,
  • providing starter materials for in-class exercises,
  • preparing your own teaching session and wanting reusable examples.
How it works

Each contributor has a personal entry in the Resources table, showing their name, an identifier such as ORCID, VIAF, or GND (when available), and a button labelled Teaching Materials.

Clicking Teaching Materials opens a list of all files shared by that contributor. Each file has a direct download link, so learners can obtain PDFs, slides, or other materials without logging into GitHub.

For users who only want to access resources, GitHub remains in the background: the interface automatically loads and displays all available materials.

How to upload your teaching materials

A detailed, step-by-step user guide with FAQs and additional instructions for uploading teaching materials is available in the GitHub repository.

1. Log in to GitHub

You need a free GitHub account. If you already have one, simply sign in.

2. Open the Resources repository

https://github.com/mano-project/mano-resources

3. Create your personal folder
  1. Click Add file → Create new file
  2. In the file name field, type your folder name followed by /README.md:
    Surname Name Affiliation ORCID/README.md (or VIAF / GND. Users with no identifier may still contribute; the identifier field is optional but recommended.)
  3. In the large text box, enter Placeholder or a brief description of yourself and your teaching area.

What it is

<MANO> provides four form-based TEI editors, one per record type: the Manuscript Editor, Person Editor, Place Editor, and Work Editor. Each turns a TEI model into an intuitive interface, letting users produce TEI-conformant records without writing XML directly, with lightweight Linked Open Data (LOD) integration. All four share the same record-metadata block (Record ID, Record title, Editor, Status, and Change note) and export a single TEI-XML file.

The Manuscript Editor follows the TEI P5 Manuscript Description module; the walk-through below uses it as the example. The Person, Place, and Work editors work the same way over their own field sets.

When to use it

Use the editors when:

  • learning or teaching the structure of TEI manuscript descriptions
  • producing manuscript metadata for coursework, workshops, or small projects
  • preparing TEI-conformant descriptions without writing XML
  • contributing records to the shared Data Collection
How it works

The Manuscript Editor is designed to make the TEI Manuscript Description module accessible to users without technical XML knowledge. All fields correspond to TEI elements, and the structure of the form helps ensure that each description is valid, complete, and standardised. Multiple manuscripts can be described in the same session using the Add Manuscript button.

The form is divided into six sections: file description, identification, content, physical description, history, and surrogates and literature. Each section collects specific types of information and, where available, can be enriched with Linked Open Data.

Throughout the form, small link icons next to field labels provide direct access to external documentation. Depending on the context, these links may open:

  • relevant sections of the TEI Guidelines
  • licence explanations (e.g. Creative Commons pages)
  • authority or help pages from external services such as Wikidata or OpenLibrary

This allows users to consult definitions, usage examples, and licence information.

Record metadata

The first block is shared by all four editors and records information about the record itself rather than its subject:

  • Record ID — the numeric id; combined with the type it forms the record URI shown beneath the field,
  • Record title — the headword/label, written to <title level="a">,
  • Editor — the responsible team member, chosen from BA.config.editors,
  • Status — unpublished, draft, or published,
  • Change note — a short description of the edit, appended to the record's revision history as a new <change> element.

These fields populate the shared TEI <teiHeader>; the record URI is also written to publicationStmt/idno[@type="URI"]. A reload button next to the Record ID refreshes the authority index (see Customization → Data and index lifecycle).

Identification

The identification section gathers core cataloging information that uniquely identifies the manuscript:

  • manuscript name,
  • collection,
  • sigle,
  • alternative identifier,
  • library (the holding institution), with geographical coordinates, settlement, and country

It corresponds to TEI element <msIdentifier> and related sub-elements.

Where available, repository and place fields can be linked to Wikidata entries. Using Wikidata URIs adds stable identifiers to the description and supports LOD practices by connecting the manuscript record to external authority data.

Content

This section describes what the manuscript contains. The user can add one or more textual items and for each item can provide:

  • author,
  • title,
  • one or more page ranges,
  • incipit,
  • explicit,
  • textual family,
  • textual subfamily,
  • text language,
  • text genre.

It maps onto TEI elements <msContents>, <msItem>, and related components.

Author and work information can optionally be aligned with authority data from Wikidata, enabling links between the manuscript description and external information about people, works, or traditions. This strengthens the LOD dimension of the metadata and supports reuse in other environments.

Physical description

The physical description section focuses on the material aspects of the manuscript. It corresponds to TEI element <physDesc> and related elements. Users can provide the following data:

  • condition description,
  • material,
  • height and width,
  • leaves, columns, and written lines,
  • number of hands and description,
  • script(s) description,
  • decoration description,
  • music notation,
  • additions,
  • accompanying material description,
  • binding description,
  • seal(s) description.
History

The history section records information about the origin and provenance of the manuscript: where and when it was produced, and how it may have travelled or changed ownership. It corresponds mainly to <history>, <origin>, and <provenance>. This section includes:

  • summary,
  • origin, it includes:
    • place of origin, together with geographical coordinates, and country,
    • date of origin, expressed both as a human-readable text and as machine-readable values.
  • one or more provenance block. Each one includes:
    • previous owner, together with geographical coordinates, settlement, and country,
    • period of ownership, expressed both as a human-readable text and as machine-readable values.

Origin places and institutions are linked to Wikidata, so that geographic and institutional information becomes part of a larger LOD network. This makes it easier to connect <MANO> descriptions with other datasets that reference the same entities.

Surrogates and literature

The final section brings together references to digital surrogates and literature references. It aligns with TEI structures for references and bibliography, such as <surrogates> and <listBibl>.

  • links to digitised facsimiles, IIIF resources, or catalogue links,
  • bibliographic entries connected to authority data where possible.

Bibliographic items can be enriched using OpenLibrary: searches in OpenLibrary help retrieve consistent book data (titles, authors, identifiers such as ISBN), which are stored together with a stable OpenLibrary identifier. This supports LOD by connecting local references in the description to a wider ecosystem of bibliographic data.

If a book is not available in OpenLibrary, users can still add the reference manually by providing the standard bibliographic fields: title, author(s), year of publication, publisher, place of publication, series, and ISBN. These manually entered records are handled exactly like imported ones, ensuring that all references remain complete and usable even when no external authority record exists.

Upload your description or Load sample

The system let the users to:

  • Upload XML

    Users may also upload an existing record created earlier in <MANO>. A previously downloaded TEI-XML file can be re-uploaded into the editor, allowing records to be inspected, modified, or expanded at any time.

  • Load sample

    It loads a complete, ready-made manuscript description that illustrates how the different fields work together.

Preview and validation

Two additional features help ensure accuracy:

  • Preview

    Generates a live TEI-XML snippet that helps learners see and understand the TEI structure behind the form.

  • Validation

    Runs validation checks to ensure required elements are present, values are well-formed, identifiers are not duplicated, and referenced items are not already in the list. This helps avoid inconsistencies when contributing records to the shared Data Collection. Validation distinguishes between:

    • Mandatory issues: errors that must be corrected before a download is permitted,
    • Optional checks: warnings about potential inconsistencies that do not prevent downloading.

    The validation ruleset lives in JS/ba/validate.js.

Output

Each editor produces a single TEI-XML file named {id}.xml, ready to be committed to the record type's folder under data/ (data/manuscripts/, data/persons/, data/places/, or data/works/). Once committed, the record is picked up by the authority index and appears in the Data Collection (see Customization → Data and index lifecycle).

What it is

The Data Collection is where the project's TEI records are published and browsed. A landing page presents four cards — Manuscripts, Persons, Places, and Works — each showing a live record count. Every record is rendered natively from its TEI file; there are no JSON intermediates.

When to use it
  • to browse the records of a given type in a filterable list,
  • to open a record's full view, rendered directly from its TEI,
  • to follow cross-links between records (for example, from a person's birth place to the place record),
  • to reach a specific record directly through a ?id= deep link.
How it works

Each per-type page loads the authority index index/index-{type}.json and lists its records. Clicking a row fetches the TEI file referenced by the index (under data/{type}/), parses it in the browser, and renders the full record — a shared header (record URI, title, status, change history) plus the type-specific fields. Cross-record references resolve against the index: when a referenced record exists it becomes an internal link, otherwise the raw URI is shown as an external link.

There is no database and no JSON record store: the pages read the same index/*.json and data/** TEI files that the editors write. Indexes are cached in the browser for one hour; the reload button on the editor pages clears that cache (see Customization → Data and index lifecycle).

How to contribute records

Create a record in the matching editor, download its TEI-XML file, and add it to the repository under data/{manuscripts|persons|places|works}/ as {id}.xml. The per-type pages carry a Contribute to Collection button that links straight to the right folder once BA.config.repoUrl is set. After the file is committed, the authority index is rebuilt (manually or by the GitHub Action) and the record appears in the Collection.

System Overview

<MANO> is implemented as a fully client-side web application. All logic—XML parsing, TEI rendering, LOD queries, validation, navigation, map visualisation, and file generation— is executed in the user's browser. The platform does not use a backend server, database, authentication layer, or stateful session storage.

This architecture is designed to remove technical barriers in teaching contexts: no installation, no licenses, and no institutional hosting are required. Learners can immediately begin creating manuscript, person, place, and work records, while instructors can reuse or contribute materials with minimal technical setup.

GitHub-Based Ecosystem

The platform uses GitHub both as a hosting environment (via GitHub Pages) and as a storage layer for its files:

  • Application code, documentation, and static assets live in this repository.
  • TEI records live in this repository under data/{manuscripts|persons|places|works}/ as one {id}.xml file each, with a generated authority index in index/*.json.
  • mano-resources: teaching materials contributed by instructors.

Records and indexes are plain static files fetched directly in the browser — no backend, no database, and no intermediary service. This serverless architecture simplifies deployment and ensures long-term transparency: all data remains visible, versioned, and accessible in the repository itself.

Linked Open Data Integration

<MANO> integrates authority data through external LOD services. These lookups are performed on-demand and client-side:

  • Wikidata API: used for persons, institutions, places, and affiliations.
  • OpenLibrary API: used for data in literature references.

Selected entities contribute both a human-readable label and a stable URI. These identifiers are included in the exported TEI-XML record, improving interoperability and aligning the records with broader cultural heritage datasets.

This section deals with extending the four TEI editors and the Data Collection. All paths are relative to the repository root.

Adding a team member

Team members are defined once in BA.config.editors (JS/ba/config.js). Append an entry:

  • id — a short xml:id (e.g. mmoliere),
  • name — the display name,
  • role — general or contributor,
  • refs — an array of authority URLs (ORCID, VIAF, and/or a staff page).

The person then appears in the shared <editor> list of the TEI header and in the Editor select of every editor. If they should also be selectable as a note author in the Person editor, add them to the noteResp vocabulary as well (see Adding controlled terminology).

Adding a field

Fields are added symmetrically across render, export, and import. Using the Person editor's occupation block as the worked example:

  1. Confirm the element/attribute (e.g. occupation/@n) exists in the relevant templates/*.xml file — the templates are the structural authority; do not edit them to invent elements.
  2. Add the input to the section's render function in the editor JS, taking every label from BA.uiText (never a literal string). For occupations this is addOccBlock() in JS/ba/person-editor.js.
  3. Extend the three functions symmetrically: getPersonData (read the value off the form), buildPersonBody (serialise it to TEI), and importPersonXML (map it back on import). The other editors follow the same get{X}Data / build{X}Body / import{X}XML naming (place-, work-, ms-editor.js).
  4. If the field is mandatory, add a rule in JS/ba/validate.js.
  5. Run the round-trip check: create a record exercising the field, export, import, export again — any diff beyond whitespace and the appended <change> element is a failure.
Adding controlled terminology

The controlled vocabularies live in JS/ba/ui-text.js under BA.uiText.vocab. Hand edits are allowed. To add a value to an existing list:

  1. Find the vocab key under BA.uiText.vocab (each key is an array of { v, l } entries).
  2. Append an entry { "v": "storedValue", "l": "Display label" } — v is written to TEI, l is the display label.
  3. Save and reload the editor.

A brand-new vocab key additionally needs its <select> wired into the editor JS (as in Adding a field, step 2).

Try it. Append { "v": "Dummy", "l": "Dummy" } to the faithValue list in JS/ba/ui-text.js, reload the Person editor, and open the Faith field — "Dummy" now appears in its dropdown. Remove the entry to revert.

Data and index lifecycle

Records live in the repository under data/manuscripts/, data/persons/, data/places/, and data/works/, one numeric {id}.xml file per record. Listings are driven by the authority index in index/index-{type}.json.

  • Rebuild the index with node tools/build-index.js, which scans data/** and writes the index/*.json files.
  • Or let CI do it: the GitHub Action .github/workflows/build-index.yml runs the same script on every push that touches data/** and commits the refreshed index.
  • Each browser caches an index for one hour; the reload button () next to the Record ID clears that cache and re-fetches all four indexes.

The Data Collection pages read exactly these files — the same index/*.json and data/** TEI records the editors produce.

Live index rebuild (optional)

By default the lookups read the prebuilt index/*.json — one request per type. A live mode can instead rebuild the index in the browser directly from the repository's data/ folder, so every lookup reflects the current files without waiting for tools/build-index.js or the CI Action. Toggle it in JS/ba/config.js:

BA.config.authorityLive = { enabled: true, ttlMs: 60000 };
  • When enabled, each lookup lists data/{type} via the GitHub API and fetches every record file, extracting headwords client-side with the same rules as tools/build-index.js. Results are cached for ttlMs (default 60 s), so a file added to data/ and pushed appears in lookups within that window.
  • This costs one listing request + one fetch per record (N+1) on each rebuild, against GitHub's unauthenticated limit of 60 requests/hour per IP. If a submit token is stored in the browser it is reused to lift that limit.
  • If the listing or any file fails to load, the lookup falls back to the prebuilt index automatically — no error is shown.
  • Recommendation: keep this OFF once the record count grows. The prebuilt index is a single request per type; live mode is N+1 and can exhaust the rate limit. Leave live mode for small collections or short-term verification that a just-pushed record is picked up.
Zotero and GeoNames configuration

Both integrations are configured in JS/ba/config.js and stay hidden until configured.

  • BA.config.zotero.apiKey — the Zotero API key. It ships in client code and is visible to every visitor, so it must be a read-only key scoped to the library.
  • BA.config.zotero.libraryType — "groups" or "users".
  • BA.config.zotero.libraryId — the numeric library id; while empty the Zotero bibliography lookup is hidden.
  • BA.config.zotero.citationStyle — the CSL style id used to format each bibliography reference (default "chicago-fullnote-bibliography"). Any id from the Zotero style repository works.
  • BA.config.geonamesUsername — a GeoNames account name; while empty the GeoNames lookup in the Place editor is hidden.

Creating a GeoNames account is not enough: the account must also enable the free web services before searchJSON returns results. Log in at geonames.org → Manage Account and click the "Free Web Services — click here to enable" link. Until this is done GeoNames answers with HTTP 200 and an in-band error object instead of results; the Place editor's lookup now surfaces that message as a red row. Common status values:

  • 10 — wrong username, or the account is not enabled to use the free web service.
  • 18 / 19 / 20 — hourly / daily / weekly credit limit exceeded.
Zotero bibliography lookup

Once a libraryId is set, every Bibliography section (in the Person, Place, Work, and Manuscript editors) shows a Search the project's Zotero library field. Type a few words of a title, creator, or year; the dropdown lists matching items as Creator (Year): short title. Selecting one fills the reference block:

  • Title of publication receives the full formatted note reference — author, title, publication details, and pages — produced by Zotero from the configured citationStyle (e.g. Ronny Vollandt, "Die aktuelle Forschung zu den arabischen Bibelübersetzungen," Theologische Literaturzeitung 144, no. 3 (2019): 147–62.). Any HTML formatting is stripped to plain text, and a trailing URL is removed while the closing period is kept.
  • Link (ptr) receives the item's Zotero URL, so the record keeps a pointer back to the library entry.

You can still edit the title by hand afterwards, or type a reference manually without using the lookup. Because the formatted note is stored as the reference's title, the Source selects elsewhere in the record (which point at these bibliography entries) show that title as their option label, truncated with the full text on hover.

Notes: the lookup needs a live network connection to api.zotero.org. If the configured citationStyle id is invalid or Zotero cannot format an item, the lookup still works but falls back to the item's short title (with a warning in the browser console). Style ids are case-sensitive and must exist in the Zotero style repository — for the current Chicago full-note style the id is chicago-fullnote-bibliography (the -17th-edition suffix belongs to the archived 16th-edition style and is rejected by the API).

Submitting records from the editor

Each editor has a Submit to repository button beside Download XML. It commits the current record to data/<type>/<id>.xml on the BA.config.github.branch branch of the repository named in BA.config.repoUrl, using the same validation gate as Download. A submit that would create a new file and one that updates an existing file are handled automatically.

Writing to GitHub requires each contributor to supply a fine-grained personal access token once, in the submit dialog. To create one:

  • Go to github.com → Settings → Developer settings → Fine-grained tokens → Generate new token.
  • Set Repository access to Only select repositories and choose this repository.
  • Under Permissions → Repository permissions, set Contents to Read and write.

The token is held in your browser's localStorage only (key ba-github-token); it is never committed to the repository and is not shared between browsers or machines. Use Forget token in the dialog to remove it. After a successful submit the lookup index rebuilds from the committed files and refreshes within about a minute.

Contributors who prefer not to store a token can use the Open GitHub's new-file page link in the dialog: it opens GitHub's web editor with the filename and record content prefilled, so the record can be committed through a normal pull request. Very large records exceed the prefill link's length limit; in that case download the file and add it on GitHub manually.
